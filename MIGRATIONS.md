# Database Migrations with Alembic

Software Builder uses [Alembic](https://alembic.sqlalchemy.org/) for database schema migrations, providing version control for database changes.

## Why Alembic?

- ✅ **Version Control**: Track all schema changes in git
- ✅ **Team Collaboration**: Share schema changes across developers
- ✅ **Rollback Support**: Downgrade to previous schema versions if needed
- ✅ **Autogenerate**: Automatically detect model changes
- ✅ **Production Ready**: Standard tool for SQLAlchemy migrations

## Quick Start

### Initialize Database (First Time)

```bash
# Create database with latest schema
just db-init
```

This creates:
- `data/software_builder.db` - SQLite database file
- All 10 model tables
- Alembic version tracking table

### Verify Current Migration

```bash
# Show current database version
just db-current

# Show all migration history
just db-history
```

## Making Schema Changes

When you modify SQLAlchemy models, create a migration:

### 1. Modify Models

```python
# src/software_builder/models/session.py
class Session(Base):
    __tablename__ = "sessions"
    
    # Add new field
    tags: Mapped[list[str] | None] = mapped_column(JSON, nullable=True)
```

### 2. Generate Migration

```bash
# Autogenerate migration from model changes
just db-migrate "Add tags to sessions"
```

This will:
- Detect changes to models
- Generate migration file in `alembic/versions/`
- Auto-format with ruff

### 3. Review Migration

Check the generated file in `alembic/versions/`:

```python
def upgrade() -> None:
    # ### commands auto generated by Alembic ###
    op.add_column('sessions', sa.Column('tags', sa.JSON(), nullable=True))
    # ### end Alembic commands ###


def downgrade() -> None:
    # ### commands auto generated by Alembic ###
    op.drop_column('sessions', 'tags')
    # ### end Alembic commands ###
```

### 4. Apply Migration

```bash
# Apply all pending migrations
just db-upgrade
```

## Common Commands

### Upgrade/Downgrade

```bash
# Upgrade to latest version
just db-upgrade

# Downgrade by one revision
just db-downgrade

# Or use alembic directly for more control
uv run alembic upgrade +1        # Upgrade by 1 revision
uv run alembic downgrade base    # Downgrade to beginning
uv run alembic upgrade <revision> # Upgrade to specific revision
```

### Migration History

```bash
# Show all migrations
just db-history

# Show current revision
just db-current

# Or use alembic directly
uv run alembic history --verbose
uv run alembic current --verbose
```

### Reset Database

```bash
# WARNING: Destroys all data!
just db-reset
```

This will:
1. Delete `data/software_builder.db`
2. Run `alembic upgrade head` to recreate schema

## Environment Variables

Override database URL with environment variable:

```bash
# Use custom database location
DATABASE_URL=sqlite:///custom/path/db.sqlite just db-upgrade

# Or export for session
export DATABASE_URL=sqlite:///custom/path/db.sqlite
just db-upgrade
```

## Configuration

### alembic.ini

Main configuration file:

```ini
[alembic]
script_location = %(here)s/alembic
sqlalchemy.url = sqlite:///data/software_builder.db

[post_write_hooks]
hooks = ruff
ruff.type = module
ruff.module = ruff
ruff.options = check --fix REVISION_SCRIPT_FILENAME
```

### alembic/env.py

Python environment configuration:

```python
# Import our models for autogenerate
from software_builder.models import Base

# Support DATABASE_URL environment variable
database_url = os.environ.get("DATABASE_URL")
if database_url:
    config.set_main_option("sqlalchemy.url", database_url)

# Set target metadata for autogenerate
target_metadata = Base.metadata
```

## Best Practices

### 1. Always Review Autogenerated Migrations

Alembic is smart but not perfect:
- Check for data loss (dropping columns)
- Verify data type changes
- Add data migrations if needed

### 2. Test Migrations Before Committing

```bash
# Apply migration
just db-upgrade

# Test your application
just test

# If issues, downgrade and fix
just db-downgrade
```

### 3. Keep Migrations Small

Create focused migrations:
- ✅ Good: "Add tags field to sessions"
- ❌ Bad: "Update all models for v2"

### 4. Never Edit Existing Migrations

Once a migration is committed and shared:
- **Don't** modify it
- **Do** create a new migration to fix issues

### 5. Data Migrations

For complex changes requiring data transformation:

```python
def upgrade() -> None:
    # Schema change
    op.add_column('sessions', sa.Column('tags', sa.JSON(), nullable=True))
    
    # Data migration
    connection = op.get_bind()
    connection.execute(
        sa.text("UPDATE sessions SET tags = '[]' WHERE tags IS NULL")
    )
```

## Troubleshooting

### "Target database is not up to date"

Migration version mismatch:

```bash
# Check current version
just db-current

# Upgrade to latest
just db-upgrade
```

### "Can't locate revision identified by 'xxxx'"

Missing migration file:

```bash
# Check history
just db-history

# Ensure all migrations are committed in git
git pull
```

### "Table already exists"

Database out of sync:

```bash
# Reset database (WARNING: destroys data)
just db-reset

# Or manually stamp current version
uv run alembic stamp head
```

### Autogenerate Misses Changes

Some changes aren't auto-detected:
- Table renames (use `op.rename_table()`)
- Column renames (use `op.alter_column()`)
- Constraint changes (sometimes)

**Solution**: Edit migration manually after generation.

## Testing with Migrations

### In-Memory Test Databases

Tests use `create_all()` for in-memory databases:

```python
@pytest.fixture
def db_engine():
    """Create an in-memory SQLite database for testing."""
    engine = create_engine("sqlite:///:memory:", echo=False)
    Base.metadata.create_all(engine)  # Direct creation, no migrations
    yield engine
    Base.metadata.drop_all(engine)
```

### Testing Migrations Themselves

To test migration up/down:

```bash
# Test upgrade
just db-upgrade

# Test downgrade
just db-downgrade

# Test full cycle
just db-reset
```
